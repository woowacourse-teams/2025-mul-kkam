# Code Style Guide

이 문서는 코드의 정답을 정의하기 위한 문서가 아니다.  
우리가 함께 코드를 읽고, 리뷰하고, 유지보수하기 위해
어떤 기준으로 코드를 판단하는지와 어떤 방식으로 사고를 전개하는지를 공유하기 위한 문서다.

## 응답과 리뷰의 태도

- 근거가 부족하거나 정보가 불확실하다면 지어내지 않는다. 모르겠다 또는 알 수 없다고 명확히 말한다.
- 답변 전에 이용 가능한 정보를 단계적으로 검증한다. 출처가 모호하거나 불분명하면 불확실하다고 표시한다.
- 검증된 신뢰 가능한 정보만으로 간결하게 답한다. 불가피하게 추측이 필요한 경우, 추측임을 명확히 밝힌다.
- 질문이 모호하거나 맥락이 부족하다면, 답변 전 추가 정보나 설명을 요청한다.
- 확인되지 않은 사실을 단정적으로 주장하지 않는다. 필요할 때는 근거를 제공한다.
- 가능하다면 출처나 참고 자료를 제시한다. 간단한 요약이나 관련 자료 링크를 함께 포함한다.
- 단계적으로 생각한다.
- 응답이 요청을 충분히 만족하지 못했다면 같은 턴 안에서 즉시 재검토하고 수정한다. 기다려 달라고 하거나 나중에 개선을 약속하지 않는다.

## 아키텍처 의존성 규칙

- 클린 아키텍처 의존성은 data -> domain <- ui 를 따른다.
- data는 ui를 import하면 안 되고, ui도 data를 import하면 안 된다.
- data와 ui 사이에는 추상화 계층을 두고 의존성 주입으로 연결한다.
- domain은 다른 어떤 레이어도 import하지 않아야 한다.

## Kotlin 코딩 규칙

- 변수나 함수 이름을 지을 때 축약어를 사용하지 않는다.
- Kotlin에서 변수를 선언할 때는 가능한 한 데이터 타입을 명시한다. 지역변수는 명시할 필요 없다.
- 하드코딩된 매직 넘버를 지양한다.
- 상수는 const val로 선언하는 것을 선호한다.
- 출력용 문자열은 별도의 리소스로 선언한다.

## 기본 철학

- 코드는 기계보다 사람이 더 많이 읽는다.
- 좋은 코드는 설명 없이도 왜 이렇게 했는지가 드러난다.
- 스타일은 규칙이 아니라 팀의 합의다.
- 다르게 썼다면, 이유가 설명될 수 있으면 된다.

## 역할과 책임

- 하나의 코드에는 적절한 주된 책임만 둔다.
- 클래스, 함수, 컴포넌트는 한 가지 역할을 설명할 수 있어야 한다.
- 여러 책임이 섞여 있다면
  - 지금은 왜 괜찮아 보이는지
  - 규모가 커지면 어떤 문제가 생길 수 있는지
  를 함께 본다.

## 네이밍

- 네이밍은 문법적 정확성보다 오해 가능성을 기준으로 한다.
- 이름만 보고 역할을 과하게 기대하게 만들지는 않는지, 반대로 중요한 책임을 숨기고 있지는 않은지를 확인한다.
- 동작 방식보다 의도를 드러내는 이름을 선호한다.
- Boolean 상태는 값 자체보다 의미가 중요하다. 왜 이 상태가 필요한지 코드만 보고 유추 가능해야 한다.

## 일관성

- 기존 코드와의 일관성을 항상 함께 본다.
- 이미 프로젝트에 쓰이고 있는 패턴, 네이밍, 구조가 있다면 그 흐름을 기본으로 따른다.
- 다르게 썼다면 의도적인 선택인지 단순한 불일치인지 설명 가능해야 한다.
- 일관성이 깨진다면 새로운 기준이 될 수 있는지도 함께 고민한다.

## Compose 스타일

- UI는 상태의 결과로 표현한다.
- 클릭, 분기, 계산 로직이 Composable 안에 직접 들어가 있다면 상태로 밀어낼 수 없는 이유가 있는지 한 번 더 본다.
- Composable의 역할을 구분한다.
  - 상태를 소유하는 컴포넌트인지
  - 상태를 전달받아 그리기만 하는 컴포넌트인지
- remember, rememberSaveable, LaunchedEffect, SideEffect 등 사이드 이펙트 API는 왜 이 위치에 필요한지 설명 가능해야 한다.
- Preview는 장식이 아니라 도구다.
  - 단순 렌더링 확인인지
  - 상태 조합 검증용인지
  의도가 분명해야 한다.

## 공통화와 확장 함수

- 공통화는 재사용성보다 읽기 흐름을 기준으로 한다.
- 확장 함수를 만드는 목적은 호출부의 의도가 더 명확해지는지에 있다.
- 파라미터가 늘어나 의미가 흐려진다면 굳이 확장 함수로 만들지 않아도 된다.
- 반복이 보이기 시작하면 추출을 고민하되
  - 지금 당장 추출할지
  - 조금 더 반복된 뒤에 추출할지
  판단 근거를 남긴다.

## 테스트

- 테스트는 구현이 아니라 행위와 시나리오를 검증해야 한다.
- 내부 구현 변경에 쉽게 깨지지 않아야 한다.
- 단위 테스트로 충분한 로직은 단위 테스트로 남긴다.

## 예외와 경계 조건

- 실패, 예외, 경계 조건은 의도적으로 다룬다.
- else로 덮거나 무시한 처리라면
  - 지금 단계에서 감당 가능한지
  - 어떤 대안이 있는지
  를 함께 고민한다.

## 문자열과 지역화

- 문자열, 날짜, 숫자 포맷은 당장 다국어가 아니더라도 지역화가 필요한 지점인지 한 번 더 본다.
- 하드코딩된 도메인 표현은 의도가 흐려질 수 있으므로 상수로 분리한다.

## 리뷰 톤

- 리뷰는 지적이 아니라 대화다.
- 단정형 표현보다 여지를 남기는 표현을 선호한다.
  - 나는 이렇게 느껴졌다
  - 이런 상황에서는 어떻게 될까
- 정답을 말하기보다 다음 선택을 돕는 것이 목적이다.
